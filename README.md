# AI Text Detection System

## Overview

This project is a Flask-based web application that estimates whether submitted text was likely written by a human or generated by AI. Rather than relying on a single heuristic, the detector combines multiple independent signals into a single confidence score and presents the result using a transparent label that explains the outcome in plain language.

The application also supports:

- Story publishing
- User verification
- Appeals for incorrect classifications
- Audit logging
- Rate limiting
- Transparent confidence reporting

The goal of the project is not to determine authorship with certainty, but to provide an explainable estimate while allowing users to challenge incorrect classifications.

# Features

- Multi-signal AI detection pipeline
- Confidence score (0.0–1.0)
- Human-readable transparency labels

# Detection Pipeline

The detector combines three independent signals.

## Signal 1 — Repetition & Vocabulary Diversity

This signal measures how repetitive a piece of writing is.

Metrics include:

- Type-token ratio
- Frequency of repeated words
- Repeated sentence beginnings

### What it captures

AI-generated text often repeats vocabulary more consistently than human writing.

### Limitation

Creative writing, poetry, and speeches intentionally reuse phrases and may appear more AI-like.

---

## Signal 2 — Sentence Structure Consistency

This signal measures sentence length variation.

Metrics include:

- Average sentence length
- Standard deviation

### What it captures

Human writing usually varies naturally between short and long sentences, while AI-generated writing often follows a more consistent rhythm.

### Limitation

Technical documentation and instructional writing frequently use uniform sentence structures despite being written by humans.

---

## Signal 3 — Agent Overview

The third signal uses an AI agent to evaluate the overall writing style.

Instead of returning a direct numeric score, the agent assigns qualitative ranges (Low, Medium, or High AI-likeness). These ranges are converted into a normalized score before being combined with the first two signals.

### What it captures

This signal evaluates characteristics that are difficult to measure using simple statistical methods, such as overall coherence and stylistic patterns.

### Limitation

Because this signal relies on a language model, it should be considered supportive rather than authoritative.

---

# Confidence Scoring

Signals are combined into a single confidence score.

Example:

```
Signal 1 = 0.65
Signal 2 = 0.03
Signal 3 = Low

↓

Final Confidence = 0.23
```

Confidence always ranges from:

```
0.0 → Likely Human
1.0 → Likely AI
```

---

# Classification Thresholds

| Confidence | Result |
|------------|--------|
| 0.00–0.39 | Likely Human |
| 0.40–0.69 | Uncertain |
| 0.70–1.00 | Likely AI |

These thresholds were intentionally chosen to reduce false positives.

---

# Transparency Labels

Instead of only displaying a number, every result includes a human-readable explanation.

## Likely Human

```
likely_human

Confidence: 0.23

The submitted text shows characteristics that are commonly found in human writing.
```

---

## Uncertain

```
uncertain

Confidence: 0.56

The available evidence is mixed. This submission should not be classified automatically.
```

---

## Likely AI

```
likely_ai

Confidence: 0.84

The submitted text contains multiple characteristics commonly associated with AI-generated writing.
```

These labels communicate the system's confidence without claiming certainty.

# Submission Endpoint

Publishing a story returns a structured JSON response.

Example:

```json
{
    "creator": "joey",
    "story_id": "827a3205-aa22-4ed7-839b-3e443347d09c",
    "attribution": "likely_human",
    "confidence": 0.236,
    "status": "classified"

```

Each response contains:

- attribution result
- confidence score
- transparency label
- story information

# Appeals Workflow

Users who disagree with a classification may submit an appeal.

Each appeal stores:

- story ID
- original classification
- confidence score
- creator explanation
- timestamp
- appeal ID

Once submitted, the story status changes to:

```
Pending Review
```

Appeals remain available for manual inspection.

---

# User Verification

Users may verify their account through the `/verify` endpoint.

Verified users receive additional trust within the application and reduce the likelihood of false allegations regarding their work.

Verification status is recorded in the audit log.

---

# Rate Limiting

Publishing is protected using Flask-Limiter.

Limits:

- 2 publish requests per minute
- 25 publish requests per day

These limits discourage spam submissions while remaining practical for normal writing activity.

When exceeded, the API returns:

```json
{
    "error": "Rate limit exceeded. Try again later."
}
```

# Validation

The detector was tested using several categories of writing:

- human-written fiction
- AI-generated stories
- edited AI responses
- repetitive passages
- technical writing

The resulting confidence scores varied as expected, demonstrating that each signal contributes differently depending on the characteristics of the input text.

---

# Known Limitations

Although the detector performs reasonably well, it is not perfect.

Examples of difficult content include:

- same scoring all around
- long text getting higher scores for signal 1
- poetry with intentional repetition
- legal documents
- technical documentation
- heavily edited AI-generated writing
- long-form narrative fiction

These writing styles may resemble AI according to one or more signals while still being entirely human made.

For this reason, the application includes an **uncertain** category and supports manual appeals.

---

# Spec Reflection

During development, the implementation diverged from the original design in one significant way.

The initial plan described a two-signal detector based entirely on statistical analysis. After implementation, a third signal was added using an AI agent to evaluate the overall writing style. Rather than replacing the original signals, it serves as an additional source of evidence that is combined with the statistical measures.

The project also expanded beyond the original requirements by adding:

- user verification
- audit logging
- structured appeal storage
- persistent story management

These additions improved transparency and made the application more representative of a real content moderation workflow.
